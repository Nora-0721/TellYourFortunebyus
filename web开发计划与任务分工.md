# Web 开发计划与任务分工

> 面向导师评审与两人协作使用，基于当前 Django + AI 面相命理 Web 项目，在**不推翻现有功能的前提下**，新增一个轻量 LLM Agent 子模块，并完善工程细节与 AI 原理学习。

---

## （一）Target（整体目标）

1. **AIWeb 应用现有模块上➕一个轻量 LLM Agent 子模块**  
   - 复用当前「上传照片 → CV 模型分析 → 命理报告」的能力；
   - 新增一个对话式 LLM Agent，让用户可以围绕自己的报告追问细节、解释原因、做延伸解读；
   - 对话支持基本历史记忆与上下文管理。

2. **项目细节完善、运行维护与测试跑通**  
   - 梳理并补齐关键视图、AI 模块的注释与文档；
   - 完善异常处理、日志记录、配置管理（API Key、模型路径等）；
   - 形成可重复部署、可基本测试的工程形态。

3. **希望我俩能理解模型的核心架构与数学原理**  
   - 对现有判别式 CNN、生成式 GAN 做结构级别的理解：输入/输出、主要层级、损失与优化思路；
   - 能手写并讲清楚关键数学概念：卷积、特征向量与相似度、GAN 的对抗目标等；
   - 形成一份「从工程到原理」的总结文档，作为课程学习成果的一部分。

---

## （二）目前情况

1. **整体架构**  
   - 基于 Django 的全栈 Web 应用：
     - 工程配置：image_upload（settings/urls/wsgi 等）；
     - 业务应用：mainsite（views/models/forms）；
     - AI 推理：face、stygan、torch_utils 等目录；
     - 模板与静态资源：templates、static。

2. **核心模型**  
   - 判别式 CNN：
     - 使用 Keras/TensorFlow 保存为 h5 文件（如 test1_mode_max_v0.h5）；
     - 主要完成「五官与面相特征 → MBTI/性格/职业推荐等」的判别任务；
     - 推理流程封装在 face_test.py / face_en.py 中，通过 cal_rate 等函数对外提供接口。
   - 生成式 GAN：
     - StyGAN/StyleGAN 相关代码存在于 stygan、torch_utils、legacy.py 等目录；
     - 用于生成「配对人像」或相关美颜/风格化效果（匹配另一半的脸等）；
     - 当前仓库主要保留推理与权重转换工具，完整训练脚本不在本项目中。

3. **已有 LLM 能力（非 agent）**  
   - 在 ai/__init__.py 中定义了 DashScopeChat：
     - 使用阿里云 DashScope 的 compatible-mode 接口，模型为 qwen-plus；
     - system 提示词为「算命大师」，用于根据几何特征/DeepFace 属性生成文案；
     - 在 face_test.py 中被用于按眼/鼻/嘴/属性生成一段算命文字；
   - 现状：
     - 仅支持「单轮调用」，没有多轮对话、对话历史管理、报告问答 UI；
     - MBTI 对应的性格文案是通过 if-else 硬编码实现的，不是 LLM 驱动的。

## （三）初步方案（初步需求和技术选型）

### A. 大语言模型应用方面

#### 1. Agent 框架选型（LangChain 版本、调用模式）

【需求】
- 基于现有 DashScopeChat 能力，扩展出一个轻量级「报告问答 Agent」：
  - 输入：当前用户的一份完整分析结果（结构化数据） + 最近若干轮对话历史 + 本轮用户问题；
  - 输出：围绕当前报告的解释/延伸解读，而不是重新算命或胡编数值；
  - 需要基本的角色约束与回答风格控制；
  - 保持工程简单，便于零基础同学理解与调试。

【技术选型】
- 框架：
  - 第一阶段：**不强依赖 LangChain**，采用「自定义小封装 + 手动 messages 管理」：
    - 继续复用 DashScopeChat 内部的 HTTP 调用逻辑；
    - 扩展支持传入完整 messages 列表（system + history + user）；
    - 由 Django view 侧维护对话历史与上下文拼接；
  - 第二阶段（可选）：在理解清楚基础逻辑后，学习性地引入 LangChain：
    - 使用 LangChain 的 ChatModel 封装 DashScope/OpenAI；
    - 使用 Memory（如 ConversationBufferMemory）管理历史消息；
    - 对照比较「手写实现」与「框架封装」的差异与利弊。

#### 2. 大模型 API 服务选型

【需求】
- 使用稳定可用、成本可控的中文大模型，支持：
  - 多轮对话接口（chat completions）；
  - system prompt、工具调用（如后续需要）；
  - 合理的响应速度与配额限制。

【技术选型】
- 优先：
  - 继续使用阿里云 DashScope（项目中已接入 qwen-plus）：
    - 统一配置 API Key（从代码硬编码迁移到环境变量）；
    - 规范化 API 调用封装，方便替换模型或服务商。
- 备选：
  - 预留 OpenAI / 通义千问其他版本等接口适配层：
    - 通过在 DashScopeChat 外再包一层通用 ChatModel 接口（如 ask_messages(messages)）；
    - 方便日后切换模型而不影响上层业务逻辑。

#### 3. 工具调用逻辑选型：Function Calling / 结构化数据读取（不打算用 RAG）

【需求】
- Agent 需要使用的「工具」其实是**已有的结构化分析结果**，而非再去调用新模型：
  - 如：MBTI 类型、眼/鼻/嘴评分、职业推荐列表、幸运色/方位、DeepFace 属性等；
  - Agent 的职责是解释和延展这些结果，而不是重新决策。
- 暂不考虑引入 RAG（向量检索），以降低复杂度。

【技术选型】
- 工具调用方式：
  - 采用「结构化数据读取 + 手工拼接上下文」模式：
    - 在 Django view 中，从数据库/缓存中取出本次分析的结构化字段；
    - 将其组织成一段结构化文本（或 JSON 序列化），放入 system / assistant 消息；
    - LLM 仅依据这些内容回答用户问题；
  - 如后续需要更复杂的「子工具」（如重新匹配职业），可再设计 function calling schema。

#### 4. Prompt engineering（系统提示词、角色定义、输出约束等）

【需求】
- 明确 Agent 的角色与回答边界：
  - 角色：负责「解释已有命理报告」，而非重新算命或生成新的数值结果；
  - 风格：温和、理性、略带玄学氛围，但避免过度迷信或恐吓式措辞；
  - 输出长度：默认 50–150 字，可按用户指令调整；
  - 安全：避免诱导性问题、避免现实世界的投资/医疗/法律建议。

【技术选型】
- 提示词结构：
  - system：
    - 明确「你是报告讲解员/算命顾问」+「只能解释已有报告结果」+「禁止输出敏感/违规内容」；
    - 内嵌 MBTI/五官结论的描述规范（可使用占位符）；
  - assistant（context）：
    - 提前提供一条「这次分析的完整摘要」，包含关键字段与简短说明；
  - user：
    - 用户提问本身。
- 输出约束：
  - 要求输出为自然语言，但鼓励特定结构（如分点回答、使用小标题）；
  - 对于不合规问题，要求用安全话术引导或拒绝。

#### 5. LLM 输出解析方案

【需求】
- 虽然主要输出是自由文本，但仍希望能：
  - 提取关键词（如提到的优点、风险点）；
  - 对异常输出（空、过短、明显不相关）做兜底处理；
  - 为未来结构化分析/可视化留接口。

【技术选型】
- 短期：
  - 要求 LLM 在文本中使用简单的标记格式（如「优点：…；建议：…」）；
  - 在后端使用正则或简单字符串处理抽取重点字段；
  - 对异常回复：
    - 检测 length 和是否包含关键字段；
    - 如不满足，则重新请求一次，或回退到本地固定模板文案。
- 中期（可选）：
  - 采用 JSON 模式输出（如 `{ "summary": "...", "advice": "..." }`）；
  - 配合 Pydantic 或手写校验器对 JSON 结构做验证与容错。

### B. Web 应用方面

#### 1. UI / 视觉与交互设计（需求 + 技术选型）

【需求】
1. 完成登录 / 注册页面重构：
   - 优化表单布局与标签/输入框对齐；
   - 完善验证码/错误提示的交互与视觉；
2. 设计并实现对话式主界面：
   - 包含图片上传区、报告摘要区、对话消息气泡区、用户输入框、发送按钮、加载状态指示器；
3. 实现图片上传预览组件：
   - 支持格式/大小校验，非人脸图片前置提示（可基于已有 dlib/DeepFace 检测结果做反馈）；
4. 统一视觉风格：
   - 现有 UI 偏古风命理 + MBTI 文案略现代，可调整为更现代、轻量的风格，减少违和感；
5. 实现响应式适配：
   - 兼容 PC 端主流分辨率与部分手机尺寸，布局在窄屏下自动堆叠，交互流畅。

【技术选型】
1. 基础技术：
   - 使用 HTML5 + CSS3 + 原生 JavaScript，为保证你同学易于上手，暂不引入重型前端框架；
2. 样式方案：
   - 在当前 static 目录基础上，拆分模块化 CSS 文件（按页面/组件划分）；
   - 如有需要，可小规模引入 Tailwind CSS 或保持 Bootstrap，统一颜色、字体和间距规范；
3. 组件库：
   - 可以继续沿用 Bootstrap 的栅格系统与基础组件（按钮、表单、模态框）减少重复开发；
4. 图标与动画：
   - 使用 Font Awesome 或类似图标库表现上传、加载、发送等状态；
   - 用 CSS transitions/animations 提升交互反馈，而不引入沉重动画库。

---

#### 2. 前后端通信与数据传输（需求 + 技术选型）

【需求】
1. 图片上传异步接口：
   - 前端通过 POST multipart/form-data 提交图片（及性别、语言等信息）；
   - 后端返回分析任务 ID 或直接返回结构化分析结果 + 命理报告图片 URL；
2. 对话消息异步发送：
   - 用户每次输入问题，由前端通过 Fetch/AJAX 发送文本 + session_id；
   - 后端调用 LLM Agent，返回回答文本与可选结构化字段；
3. 推理耗时处理：
   - 如 AI 推理耗时较长，可采用轮询任务状态或分阶段返回（可按时间与工作量评估）；
4. 统一错误处理：
   - 覆盖网络错误、超时、图片无效、模型推理失败等场景；
   - 前端展示友好错误提示，不泄露内部异常细节；
5. 请求防抖与防重提交：
   - 发送请求期间禁用按钮或加入防抖逻辑，避免重复分析或重复提问。

【技术选型】
1. 通信方式：
   - 使用原生 Fetch API 实现异步 HTTP 请求；
   - 如有需要，可封装一个轻量 request 工具函数统一处理 headers/错误；
2. 数据格式：
   - 结构化数据均使用 JSON；
   - 图片上传使用 multipart/form-data，后端使用 Django 的文件上传机制处理；
3. 接口规范：
   - 遵循 RESTful 风格：
     - 如 POST /api/analyze-face/、POST /api/report-chat/；
   - 返回体统一为 `{ "code": 0 | 非 0, "msg": "...", "data": {...} }`；
4. 超时与重试：
   - 在前端设置合理超时时间；
   - 对于网络波动类错误，视需要实现有限次自动重试或「重试」按钮。

#### 3. 前端状态与对话历史管理（To Do + 技术选型）

【To Do】
1. 前端对话历史存储：
   - 在内存中维护消息队列，区分 user/assistant 角色，记录时间戳与内容；
2. 对话历史渲染：
   - 按时间顺序渲染气泡式消息列表，支持自动滚动到最新一条；
3. 会话重置功能：
   - 提供「新对话」按钮，清空当前历史，并通知后端开启新 session；
4. 页面刷新后的短暂恢复（可选）：
   - 通过 sessionStorage 存储最近一段对话，实现基本的会话恢复。

【技术选型】
1. 状态管理：
   - 使用简单的 JS 对象/数组维护前端状态，通过事件驱动更新 DOM；
   - 若日后引入 Vue/React，可迁移到 Pinia/Vuex/React Context 等状态库；
2. 持久化方案：
   - 使用 sessionStorage 存储当前 session_id 与最近 N 条消息；
   - 避免长久存储敏感数据，不使用 localStorage 长期保存；
3. 消息数据结构：
   - 统一为 `{ role: 'user' | 'assistant', content: string, timestamp: number }`，便于前后端统一处理。
