# 理想伴侣预测 & 生图集成方案（lcf v0）

> 分支：lcfv0（基于 init / teacher original version）  
> 负责人：lcf（理想伴侣 LLM 输出与生图衔接）

## 1. 目标概述

在现有 `cal_rate` + 通义千问单器官算命的基础上，引入一个新的「理想正缘」LLM 输出层：

- 面向用户：生成一段中文的理想伴侣/正缘性格与相处模式的算命风格报告（展示在 Web 报告第二页）。
- 面向生图：生成一小段**纯中文短语**构成的外貌与气质提示词 `visual_prompt`，供国内文生图 API（优先阿里/腾讯）使用，用于生成“理想伴侣人像”。
- 前端交互：报告页左右滑动分页展示——第一页为本人报告，第二页为正缘报告 + 生成的人像头像；PDF 长图后续迭代。

## 2. 数据与现状梳理

### 2.1 现有本人报告流程

1. 用户上传头像 → `mainsite/views.py` 保存到 `PostNew` 等模型。
2. 调用 `face/face_test.py: cal_rate(file_name, sex)`：
   - 提取 68 关键点；
   - `judge_eye/judge_nose/judge_lip` 分别做五官分析（可选调用 `DashScopeChat` 补文案）；
   - `judge_MBTI` 给出 MBTI/性格文案与职业推荐；
   - 返回一个包含图片路径和文本分析结果的列表（约 19 项）。
3. `mainsite/views.py` 调用 `face/generate_report.py: generate_report(...)`：
   - 将分析结果写入一张长图（PNG），作为“本人报告图”。
4. 模板 `templates/showpage*.html` 渲染报告图与文字摘要。

### 2.2 现有匹配/配对功能（非理想伴侣）

- `face/face_test.py: cal_pipei()`：
  - 使用 GAN 随机生成两张配对人像 + 匹配指数和祝语；
  - 逻辑偏娱乐，未真正基于用户性格推演理想伴侣。
- 前端模板 `showpage_pipei.html` 展示两个头像 + 祝语。

### 2.3 现有 LLM（通义千问）接入

- `ai/__init__.py: DashScopeChat`：
  - 使用 `DASHSCOPE_API_KEY` 调用通义千问兼容模式接口；
  - 当前仅提供 `ask(message: str) -> str` 单轮调用。
- 调用点：`judge_eye/judge_nose/judge_lip` 中用于生成单个部位算命话术，无“综合人格+理想伴侣”的 Agent。

## 3. 新能力设计：理想伴侣预测

### 3.1 理论与话术依据（简要）

- 面相维度：额头、眼、鼻、嘴、下巴等象征性格倾向、情绪表达、稳定度（只抽象要点，不做迷信推理）。
- 阴阳互补 + 同频：
  - 情绪上：一方情绪波动大，则正缘更稳、更具安抚感；
  - 性格上：内向者匹配略外向/善社交的人，外向者匹配能倾听、给空间的人；
  - 价值观上：对成长、稳定、自由等方向有相似追求。
- 合婚/姻缘常见话术风格：
  - “相互成就”“彼此是生命中的助力/贵人”；
  - “缘分不急不缓，多在你某些转折期出现”；
  - “对方的出现，会让你看见自己某一面的成长（安全感、责任感、柔软度等）”。

### 3.2 LLM 输出 JSON Schema

LLM 每次只返回一个 JSON，对外约定结构：

```jsonc
{
  "display_text": "string, 中文正缘报告正文，300-450 字，3-5 段，命理+心理咨询风格",
  "partner_traits": {
    "keywords": ["温柔", "理性", "外向", "重家庭"],
    "interaction_style": "偏温柔但有主见，愿意主动沟通和安抚你的情绪",
    "growth_direction": "会鼓励你在保持稳定的前提下，多尝试新事物"
  },
  "partner_profile": {
    "profession_type": "如：互联网/教育/艺术/自由职业等笼统类型",
    "life_rhythm": "如：工作日较忙、周末有自己的兴趣爱好",
    "financial_style": "如：理性消费、重视基本储蓄，但不会一味省钱"
  },
  "visual_prompt": "string, 约25-60字的中文短语，逗号/顿号分隔，仅描述外貌与画面风格"
}
```

约束：

- **只允许这几个字段**，禁止新增字段或自然语言包裹（如“以下是 JSON：...”）。
- `display_text`：
  - 中文；
  - 300–450 字以内，3–5 段（用换行分段）；
  - 风格为“命理话术 + 温和心理咨询”，积极、不绝对、不贴负面标签。
- `visual_prompt`：
  - 仅使用中文；
  - 由若干短语构成，使用逗号或顿号分隔；
  - 总长度约 25–60 个汉字；
  - 只描述可视化信息：性别、年龄段、东亚脸、脸型、肤色、眼睛、鼻子、嘴唇、头发、妆容、气质、画面/光线/风格；
  - 不包含“贵人运、桃花旺、事业运”等抽象算命词。

### 3.3 LLM Prompt 设计思路

- System Prompt（示意）：
  - 你是一位结合中国面相传统与现代心理学的情感咨询师，用温和积极的方式，为用户推演“适合他的长期伴侣/正缘类型”。
  - 基于输入的性格分析、MBTI 倾向、五官象征含义等，推演出一位与用户【价值观同频、性格互补、情绪更稳定】的长期伴侣形象。
  - 输出只允许 JSON，不得出现额外解释文字。
  - 严格控制字数：`display_text` 300–450 字；`visual_prompt` 25–60 个汉字。
  - `visual_prompt` 只使用中文短语、逗号/顿号分隔，且只描述外貌特征与画面风格。

- User Prompt（示意）：
  - 向 LLM 提供：
    - 用户性别、年龄段；
    - MBTI 或类似性格标签（如“偏理性/偏感性、偏内向/偏外向、重事业/重家庭”等）；
    - 若干五官/运势关键词（如“眼神温和但略显疲惫、嘴唇较薄、当前处于事业打拼期”等）；
  - 要求其按 JSON Schema 返回结构化结果。

## 4. 文生图提示词策略（国产模型）

### 4.1 Prompt 统一方案

- **主语言**：只使用中文。
- **格式**：若干中文短语 + 逗号/顿号分隔。
- **长度**：约 25–60 个汉字。
- **内容顺序示例**：
  - 人物基础：东亚人 / 中国人、性别、年龄段；
  - 脸与五官：脸型、肤色、眼睛、鼻子、嘴唇、下巴；
  - 头发与妆容：发色、长度、发型、淡妆/日常妆；
  - 气质与风格：温柔气质、书卷气、元气感、干练等；
  - 画面与光线：写实风格、近景人像、柔和自然光等。

示例（供 LLM 学习，不直接抄给用户）：

- “东亚人、25-30岁女性、瓜子脸、皮肤白皙、长直黑发、大杏眼、自然双眼皮、淡妆、温柔气质、写实风格、柔和自然光”

### 4.2 对接形式

- 后端暴露统一接口给生图实现方（刘景琦）：
  - 输入：`prompt: str`（即上面 `visual_prompt` 字段内容）；
  - 输出：`image_url: str`（或静态路径）。
- 你这边负责：从 LLM JSON 中取出 `visual_prompt`，传给这个接口即可。

## 5. 后端改造计划（不含具体代码细节）

### 5.1 LLM 封装

- 在 `ai/__init__.py` 中新增一个高层封装函数，例如：
  - `generate_ideal_partner_profile(user_context: dict) -> dict`：
    - 负责组装 system + user prompt；
    - 调用 DashScopeChat；
    - 解析 JSON 字符串为 Python dict，做基本字段校验；
    - 返回包含 `display_text`、`partner_traits`、`partner_profile`、`visual_prompt` 的 dict。

### 5.2 调用链接入位置

- 候选位置 A：`face/face_test.py: cal_rate()` 内部在完成 MBTI 和五官分析后，补充一次 LLM 调用，将结果打包进返回结构（例如增加一个 `ideal_partner` 字段）。
- 候选位置 B：`mainsite/views.py` 中，在拿到 `cal_rate` 的结果后独立调用 LLM（不污染原有 AI 推理代码），优点是耦合更低。

当前建议：**优先使用 B 方案**——在视图层做一次 LLM 调用，把 `cal_rate` 的输出整理成 `user_context`，交给 `generate_ideal_partner_profile`，然后将返回结果存入数据库或直接传给模板。

### 5.3 数据库存储

- 在 `mainsite/models.py` 中为 `PostNew` 或相关结果表增加字段（示意）：
  - `ideal_partner_json`（TextField / JSONField）：存整个 JSON；
  - `ideal_partner_text`（TextField）：冗余存一份 `display_text`，便于直接渲染；
  - `ideal_partner_prompt`（CharField）：存 `visual_prompt`；
  - `ideal_partner_image_url`（CharField）：生图生成后回写的 URL。

## 6. 前端展示与交互规划

### 6.1 Web 报告页

- 第 1 页：现有本人报告（基本不动，只适度压缩文字长度即可）。
- 第 2 页：理想伴侣/正缘报告：
  - 标题：如“你此生的关键缘分”；
  - 文本：`display_text`（首屏可只展示前 250–300 字，点击“展开”查看完整 450 字内容）；
  - 头像：生图生成的人像 `ideal_partner_image_url`；
  - 交互：只提供“向左返回本人报告”的按钮或滑动手势，不再向右继续翻页。

### 6.2 PDF 报告（后续阶段）

- 当前版本仅规划、不实现：
  - 方案 A：PDF 中追加一整页“正缘报告 + 头像”长图；
  - 方案 B：在现有长图右侧加栏展示正缘内容。
- 等 Web 端体验稳定后，再统一设计 PDF 导出格式。

## 7. 本地测试与联调思路

### 7.1 本地 LLM 测试

- 在 `python manage.py shell` 中：
  - 调用现有 `cal_rate`，构造 `user_context`；
  - 调用新封装的 `generate_ideal_partner_profile(user_context)`；
  - `print(json.dumps(result, ensure_ascii=False, indent=2))`，重点检查：
    - `display_text` 是否合规、无明显负面/歧视用语；
    - `visual_prompt` 是否为短语列表、长度与内容合理。

### 7.2 生图测试

- 在生图接口实现后，使用几组典型 `visual_prompt`：
  - 不同性别、年龄段、气质风格；
  - 调用选定的国产文生图 API（如通义图像、腾讯混元），生成多张图；
  - 人工评估：是否明显为东亚人脸、是否与描述大致一致、有无畸形或不适宜内容；
  - 迭代优化 `visual_prompt` 模板与 LLM 规则。

### 7.3 全链路联调

- 在测试环境跑完整流程：上传图片 → 本人报告 → 正缘报告 + 头像。
- 检查：
  - LLM 和生图调用的延迟是否可接受；
  - 异常情况下的降级提示（如 LLM 或生图失败时仍能正常展示本人报告）。

## 9. 与生图后端及前端的对接约定（给刘景琦）

1. **后端函数级接口约定**（推荐）：

   - 函数签名：`generate_match_face(prompt: str) -> str`
   - 输入：
     - `prompt`：直接使用 LLM JSON 中的 `visual_prompt` 字段，形如：“东亚人、25-30岁女性、瓜子脸、长直黑发、大杏眼、自然双眼皮、淡妆、温柔气质、写实风格、柔和自然光”。
   - 输出：
     - `image_url`：生成的人像图片访问路径，可以是：
       - 形如 `/media/match_faces/xxx.png` 的相对 URL；
       - 或完整的 http(s) URL（如走对象存储/CDN）。
   - 错误处理：
     - 生图失败时，返回空字符串 `""` 或一个占位图 URL，并在后端记录日志；
     - 不抛出异常到上层视图，避免影响本人报告展示。

2. **HTTP 接口约定**（如不方便直接函数调用）：

   - URL 示例：`POST /api/generate_match_face/`
   - 请求体（JSON）：
     ```json
     {
       "prompt": "东亚人、25-30岁女性、瓜子脸、长直黑发、大杏眼、自然双眼皮、淡妆、温柔气质、写实风格、柔和自然光"
     }
     ```
   - 响应体（JSON）：
     ```json
     {
       "image_url": "/media/match_faces/ideal_partner_123.png"
     }
     ```
   - lcf 侧只负责：从 LLM 结果中取出 `visual_prompt`，组织上述请求，并拿到 `image_url` 存库或传模板。

3. **模板与前端对接点**：

   - 在视图中，统一向模板传入以下字段：
     - `ideal_display_text`：LLM 输出的正缘报告正文（`display_text`）；
     - `ideal_visual_prompt`：供生图的提示词（`visual_prompt`，可选在前端调试时展示）；
     - `ideal_partner_image_url`：生图生成的理想伴侣头像 URL。
   - 前端第二页（正缘页）布局中，至少预留：
     - 文本区域：展示 `ideal_display_text`（可折叠/展开）；
     - 图片区域：展示 `<img src="{{ ideal_partner_image_url }}">`；
     - 交互：返回第一页的按钮/手势。

4. **联调流程建议**：

   1. 第一阶段：仅联通 LLM 部分，终端打印 `ideal_display_text` 与 `ideal_visual_prompt`，不接生图；
   2. 第二阶段：由刘景琦实现 `generate_match_face` 并在视图中调用，将返回的 `image_url` 打印在终端，同时写入模板；
   3. 第三阶段：前端根据上述三个字段完成第二页展示与交互；
   4. 第四阶段：一起压测与调优调用耗时、失败降级等细节。
