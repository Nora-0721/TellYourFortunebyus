# 项目情况与改进方向说明

> 面向：你（偏熟练、懂前端/LLM），以及一位零基础同学，用来快速理解这个「AI 看面相 + 命理报告」项目现在的状态和后续可改进方向。

---

## 一、当前项目整体情况

### 1. 技术栈与目录结构

- 后端框架：Django 2.x
  - 核心配置在 [image_upload/settings.py](image_upload/settings.py)
  - 业务应用在 [mainsite](mainsite) 目录
- 数据库：SQLite，本地文件 [db.sqlite3](db.sqlite3)，适合开发环境和小规模部署。
- 缓存：Redis
  - 在 [image_upload/settings.py](image_upload/settings.py#L62-L75) 中配置了 `django_redis`，用于验证码/登录态等缓存。
- AI / CV 模型：
  - 主逻辑集中在 [face](face) 目录：
    - 推理与报告绘制脚本 [face/main_get_results.py](face/main_get_results.py)
    - 模型调用与特征分析 [face/face_test.py](face)（未在这里展开，但被多处引用）
    - 报告绘制 [face/generate_report.py](face/generate_report.py)
  - 模型文件：如 [test1_mode_max_v0.h5](test1_mode_max_v0.h5)、若干 `.npy`。
- Web 层：
  - 路由和视图： [mainsite/views.py](mainsite/views.py)、[mainsite/urls.py](mainsite/urls.py)、[image_upload/urls.py](image_upload/urls.py)
  - 模板： [templates](templates) 目录中已有首页、输入页、展示页（PC + 手机）等多种 HTML 模板。
  - 静态资源： [static](static) 目录中有 CSS / JS / 图片等。

从结构上看，这个项目已经是一个「可跑起来的 Django Web + AI 推理」原型，而不是从零开始的新项目。

---

### 2. 业务流程现状

从 [mainsite/views.py](mainsite/views.py) 和模板可以梳理出当前的典型流程：

1. **登录 / 鉴权**
   - 视图：`loginMy`、`login_bak` 等（在 [mainsite/views.py](mainsite/views.py#L39-L91)）。
   - 支持：
     - 账号密码登录（`loginMy` 走 Django 内置认证）。
     - 搭配 Redis 的验证码 / 临时 code 登录（`login_bak` + `get_code`）。

2. **上传图片（录入样本）**
   - 模型：`PostNew`、`PostNewImage` 等（在 [mainsite/models.py](mainsite/models.py) 中定义）。
   - 上传后会写入数据库，包含：图片路径、性别、语言等信息。

3. **调用 AI 模型进行面相分析**
   - 关键函数：`cal_rate`（从 [face/face_test.py](face/face_test.py) 引入）。
   - 在首页视图 `home_page_ch` / `home_page_en` 等中调用：
     - 从数据库中取最近一条上传记录 `PostNew.objects.latest('pub_date')`。
     - 根据性别选择配对图库路径，例如 `/static/meinv/`、`/static/meinan/`。
     - 调用 `cal_rate('.' + post.cover.url, int(post.sex))` 获取一系列分析结果（五官特征、整体评价、职业建议、幸运色、匹配度等）。

4. **生成命理报告图片**
   - 核心函数：`generate_report`（从 [face/generate_report.py](face/generate_report.py) 引入，或者旧版在 [face/main_get_results.py](face/main_get_results.py) 里有一份类似逻辑）。
   - 作用：
     - 加载一张背景图（如 `./static/picture/report.png`）。
     - 将用户头像裁剪为圆形后贴到报告上。
     - 写入幸运颜色、幸运方位、推荐职业、五官分析文字、综合评价等。
     - 贴上两张「像你对象的明星脸」配图及吻合度。
     - 输出一张整合后的长图（报告）。

5. **前端展示**
   - 页面模板：
     - 中文 PC：`showpage.html`
     - 英文 PC：`showpage_en.html`
     - 手机端：`show_phone.html` / `show_phone_en.html`
   - 视图片段示例：`home_page_ch` 在生成报告后，通过 `template.render(locals())` 把结果和图片 URL 渲染到页面。

整体来看：**从上传一张脸，到生成带命理分析的报告图片，再到网页展示，整条链路已经贯通**，只是还有不少「工程化」「产品化」上的改进空间。

### 3. AI / 报告部分的现状

结合 [face/main_get_results.py](face/main_get_results.py)：

- 已有比较完整的「报告绘制」逻辑：
  - `drawCircleAvatar`：把头像裁成圆形叠加到背景图。
  - `insert_word` / `insert_word_recommand`：控制多行文本在图片上的排版。
  - `generate_report`：组合背景、头像、二维码、文本、配对人像，生成最终 PNG。
- 存在的一些问题：
  - 字体路径写死在开发者本地路径，如：`/Users/jacky/Desktop/Tellyourfortune_Project/face/simhei.ttf`，不利于跨环境部署。
  - 部分逻辑在 Django 视图里也有一份类似实现，存在一定重复；可以考虑抽象成统一的工具函数。

对你来说，AI 侧的「模型调用和结果结构」基本已经封装好了，你更多是对接与包装；对你同学来说，可以通过这个部分理解「如何在 Python 中处理图片、文字排版」。

## 二、存在的问题与改进方向

这一部分更偏「工程/产品层面」，是你们后续 Web 开发可以发力的点。

### 1. 架构与接口层

- **问题 / 现状**：
  - 目前大部分逻辑写在 Django 同一个 `views.py` 里，函数较多，耦合度高，新手阅读压力大。
  - AI 推理函数直接在视图里调用，前端是同步等待，缺少明确的 RESTful API 约定。
  - 登录与验证码逻辑相对散乱（`loginMy` + `login_bak` + Redis + npy 文件），不易扩展。

- **改进方向**：
  - 把「AI 分析」封装为独立 API：例如 `/api/face-analysis/`，输入为图片/图片 URL，输出为 JSON 结构的分析结果和报告图 URL。
  - 拆分视图：按功能拆成多个模块（例如 auth_views.py / face_views.py / admin_views.py），方便你同学只看「一个模块」学习。
  - 定义清晰的接口文档（可以用 markdown 写），便于前端和移动端统一调用。

### 2. 前端体验

- **问题 / 现状**：
  - 模板众多但风格偏老（基于简单 Bootstrap + 静态资源），交互也主要是同步刷新，缺少「上传后进度提示 / 异步结果轮询」。
  - 移动端已有，但老师希望有一个 Web 版本，你可以在现有模板基础上优化或重写。

- **改进方向**：
  - 采用你熟悉的前端技术栈（例如 React/Vue 或者较现代的纯 HTML + fetch）：
    - 独立一个「上传与结果页」：上传后立即返回「任务已创建」，前端通过轮询/长轮询拿结果。
    - 把分析结果结构化展示（除了报告图片，还可以有表格/卡片形式的文字说明）。
  - 渐进式改造：保持旧模板可用，让你同学从修改现有 HTML/CSS 开始练手，再逐步引入简单 JS。

### 3. 部署与配置

- **问题 / 现状**：
  - settings 中 `DEBUG=True`，`SECRET_KEY` 明文，适合教学/开发，但不适合上线。
  - 字体路径、本地路径写死在代码里（如 `simhei.ttf` 的绝对路径）。
  - 依赖环境（TensorFlow/Keras/dlib 等）较重，对部署同学不太友好。

- **改进方向**：
  - 把所有路径和开关迁移到配置：
    - 使用 `settings.py` 里的常量或 `.env` 文件，统一管理模型路径、字体路径、静态路径。
  - 补充一份「部署说明」：列清楚 Python 版本、依赖安装、Redis 启动方式、迁移命令等。
  - 如果你有精力，可以用 Docker 做一个开发/部署环境，让你同学只用 `docker-compose up` 就能跑起来。

### 4. 代码可读性与新手友好

- **问题 / 现状**：
  - 对零基础同学来说，当前代码一次性信息量很大：Django + Redis + AI 模型 + 图片处理混在一起。

- **改进方向**（专门考虑你同学的学习路径）：
  - 把项目拆成「学习任务」：
    1. 先从模板和 CSS 开始：修改一个展示页面的文字、样式，让她先有「能看到成果」的成就感。
    2. 再让她接触最简单的视图函数：接收一个表单，返回一个页面（不含 AI）。
    3. 最后再把她拉进「调用已经写好的 `cal_rate`/`generate_report`」这一步，侧重理解「输入/输出」而不是模型细节。
  - 在关键视图/函数上方写简短中文注释（你来写），帮助她对照理解。
